# LLM Context Translation Design

状态：需求已确认，待进入实现规划。

## 目标

将增强型 LLM 翻译作为本 fork 的核心默认能力：先理解完整源字幕并统一疑难术语，再由主翻译角色生成译文，最后由独立高级校对角色完成质量审计。同时保留非 LLM 翻译和原有单模型 LLM 翻译，两者不受增强流程影响。

技术 schema、token 规划、缓存、重试、迁移和控件细节见 [Implementation Specification](./llm-context-translation-implementation.md)。

## 三种平行翻译方式

1. **非 LLM 翻译**：继续使用 Bing、Google、DeepLX 等服务，不进入全文分析、术语确认或翻译质量审计。
2. **单模型 LLM 翻译**：复用主翻译模型与主翻译 Prompt 直接分批翻译；保留原有“反思翻译”，但不获得增强型工作流承诺的双角色复核。
3. **增强型 LLM 上下文翻译**：依次执行全文分析、双角色术语裁决、主模型翻译和高级校对审计。

增强型是新安装和无历史选择时的默认方式，但不显示“推荐”徽标。用户明确选择其他方式后沿用其选择，不暗中执行增强步骤。升级时旧 LLM 翻译迁移为增强型并将原模型方案一次性绑定到两个角色；旧非 LLM 选择保持不变。

三种方式是产品入口、配置作用域和运行行为上的平行工作流；实现可共享字幕读写、批处理、网络客户端和确定性校验基础设施。

## 增强型工作流

1. 任务创建时冻结主翻译、高级校对两个角色的模型方案与独立用户 Prompt。
2. 主翻译角色分层覆盖全部源字幕，汇总内容大纲、背景、主题和影响译法的表达特征，形成翻译上下文简报；长内容按 token 分窗并递归汇总，不强求一次装入全文。
3. 主翻译角色从全文中提取存在实质翻译疑问或一致性风险的疑难术语候选。相同词形按义项处理，无法确信同义时保守拆分。
4. 主翻译角色先提出术语译法；高级校对角色看到主建议后依次判断候选是否为术语，并作出接受、纠正或首轮不确定的裁决。它不是两个模型并列投票。
5. 首轮不确定时补充更多代表语境进行一次最终校对。机械响应错误重试耗尽后，仅该术语保留源词并标记高风险，任务继续。
6. 默认自动采用高级校对形成的最终译法。交互式 GUI 任务可在开始前选择人工确认并在该阶段主动暂停；CLI、批量和非交互 workflow 始终自动确认。
7. 确认结果形成任务级权威术语表，并自动保存为 `【项目术语表】<文件名>.vcglossary.json`。精确匹配的术语表可在未来任务中跳过术语提取和裁决，但仍执行全文分析。
8. 主翻译角色按 token 预算分批翻译全部字幕。每批主体携带前后各最多 3 个只读边界语境段，主体和语境严格分离，输出只能包含主体 ID。
9. 高级校对角色分批覆盖全部源文与译文，检查翻译质量，并为每条问题字幕给出唯一、完整、已合并全部修正的建议译文，不再交回主翻译角色。
10. 翻译模块产出初版字幕、权威术语表和翻译审计报告。完整 workflow 随即把初版字幕交给独立的【字幕后处理】阶段。

逻辑上共三遍全文覆盖：主模型分析、主模型翻译、高级校对质量审计。术语校对只读取候选代表语境，不额外扫描全文。

## 术语确认

- 自动术语确认是默认值，不阻塞任务，并在报告中记录选择来源和校对回退。
- GUI 人工确认进入全尺寸主从式工作台：左侧为术语列表，右侧展示主建议、校对结论、最终译法和代表语境。
- 页面不维护逐项待确认状态。每项初始采用高级校对结果，用户可直接继续，也可改选主建议、自定义译法、合并候选或忽略。
- 系统相信用户选择，不用空自定义值等输入阻止继续，也不自动恢复默认译法。

## 翻译质量审计

- 审计覆盖语义忠实、漏译／增译、原文照抄、关键数字与符号、否定情态、人物称谓与指代、权威术语与新疑难术语、跨段连贯、目标语质量和格式保真。
- 时间轴、CPS、断句、段落合并拆分、排版、间隙、尾部补偿和通用标点清理属于【字幕后处理】，不进入翻译审计。
- 审计策略只有两种：**审计并人工确认**和**自动采纳校对修正**。GUI 默认自动采纳，也可在独立字幕任务中切换为人工确认；CLI、workflow 与批量任务始终自动采纳。
- 自动模式应用高级校对给出的全部有效修正版，包括语义、术语和目标语言质量问题；本地确定性校验只阻止数字、URL、占位符、标签、权威术语或换行结构遭到破坏，不再按“客观／主观”类别限制高级校对职责。
- 人工确认页允许逐条采纳、保留原译文或一键采纳全部有效建议，并在最终字幕写回前完成选择；无需维护跨任务的待确认状态。
- GUI 独立字幕翻译任务可进入全尺寸确认／结果页；workflow、CLI 和批量任务只保存报告并继续，不因展示阻塞。
- 报告按字幕记录原译文、校对建议、最终译文及自动采纳、用户采纳、用户保留或校验失败等处理结果。
- GUI 独立任务、GUI workflow、CLI 和批量任务均在输出目录生成 `【翻译审计】<文件名>.md`。

## 模型与 Prompt

- 模型配置方案由用户命名，保存接口类型、供应商方言、Base URL、API Key、模型名称、工作上下文上限和最大并发；选择器只显示方案名称。
- 主翻译与高级校对是独立角色，使用独立用户 Prompt，但允许绑定同一模型方案。系统不自动复制用户新填写的单个角色配置。
- 仅配置主翻译角色时，单模型 LLM 可用、增强型不可用；仅配置高级校对角色时，两种 LLM 方式均不可用。
- 两个角色各有一个持久化当前方案绑定。设置页与任务页若都展示选择控件，则读写同一个值；任务创建时冻结完整快照，运行中设置变化只影响未来任务。
- 工作上下文上限代表用户希望采用的单次工作预算，而非模型技术极限，默认 64k tokens。旁边提供手动能力探查按钮，结果只作参考，不自动覆盖用户值。
- Prompt 顺序固定为：系统硬约束 → 用户自定义角色 Prompt → 翻译上下文简报 → 当前调用相关权威术语 → 阶段指令 → 动态字幕。

## 接口与缓存原则

- 传输层支持 OpenAI-compatible、Anthropic Messages 和 Gemini，并与通用、OpenAI、DeepSeek、Kimi、GLM、Qwen 等供应商方言分离。
- 旧配置保持通用方言；用户手动选择供应商，不根据 URL 或模型名称自动识别。
- 缓存完全隐式。系统保持稳定 Prompt 前缀并自动使用供应商原生缓存能力；不可用时退回普通请求，不阻断翻译。
- 不使用对话 fork 或 `previous_response_id` 规避输入费用；动态字幕块保持独立请求和并行能力。

## 失败、取消与模块边界

- 可恢复的瞬态传输错误和机械响应错误只在安全的当前调用／字幕块边界重试；永久配置错误不重试。
- 必需阶段整体失败时，增强任务失败并停止当前 workflow，保留已有中间产物；不得自动降级为普通 LLM，也不得把未经完整审计的字幕交给后处理。
- GUI 批量任务隔离当前项失败并继续下一项。
- 取消时停止派发新请求并丢弃仍在途的结果；任务标记已取消，不产出初版字幕、不进入后处理，已正式保存的项目术语表可以保留。
- 通用任务重启恢复、全软件统一错误码、跨模块重试、队列熔断和失败重跑不在本次范围，后续作为独立任务错误处理系统设计。
- 本设计只修改字幕翻译及其必要的模型配置、任务编排和 UI；术语表和翻译审计逻辑不得延伸到【字幕后处理】。

## 设置与入口

- 设置侧边栏保留一个「翻译设置」，内部使用“非 LLM 翻译｜LLM 翻译｜增强型 LLM 翻译”三段式分页；分页只是导航，不改变任务实际方式。
- 任务入口用三张并列卡片选择方式。缺少配置的卡片仍可选中并说明缺失项，但禁用开始操作并提供设置入口。
- 单模型 LLM 入口显示主模型、主 Prompt、反思翻译和批处理大小；增强型入口显示两个角色、两份 Prompt、上下文、术语确认、审计策略和术语表导入，不显示反思翻译。
- 单模型与增强型 LLM 各自保存独立的 `batch_size`；增强型的值只限制正式翻译主体数量并允许按 token 自动缩小。全文分析、术语处理和审计按 token 自动分批，不增加各自批量设置。最大并发属于模型方案。

## Usage 统计

- 任务摘要和审计报告按阶段、按角色记录调用次数、输入／输出 token，以及供应商实际返回的缓存读写 token。
- 接口未返回的字段显示不可用，不自行估算。第一版不计算货币金额。
